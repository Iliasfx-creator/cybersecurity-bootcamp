# Stored XSS Notes

## What stored XSS is

Stored XSS happens when an application stores user-controlled input and later renders it in a page without safe output encoding.

The payload persists in the application instead of existing only in one request.

## How it differs from reflected XSS

Reflected XSS appears immediately in the response to the same request.

Stored XSS is submitted first, stored by the application, and rendered later.

This makes stored XSS more dangerous because future viewers can be affected.

## Source, storage, and sink

The source is where input enters the application.

In this lab, the source was the comment field in the POST /comment body.

The storage location was the comments list in server memory.

The sink was the HTML response generated by GET / when rendering comments.

## Why persistence matters

Persistence matters because the payload remains available after submission.

A user can submit the payload once, and the application can keep serving it to later viewers.

This can affect people who never interacted with the original malicious request.

## Output encoding

Output encoding converts special characters into safe text representations before data is placed into a response.

For example:

- < becomes &lt;
- > becomes &gt;

This prevents the browser from interpreting stored input as HTML or JavaScript.

## Why escaping at render time matters

Escaping at render time matters because the correct escaping depends on where the data is being used.

In this lab, the data was rendered in an HTML body context, so html.escape was used.

Stored data should not be trusted just because it is already inside the application.

## Why filtering input alone is risky

Filtering input alone is risky because filters can miss encodings, alternate syntax, or different output contexts.

A safer approach is to combine reasonable validation with context-aware output encoding.

The most important point is to prevent user-controlled data from becoming executable code in the browser.

## Production defenses

Production defenses include:

- context-aware output encoding
- secure template engines with autoescaping
- input validation
- content security policy
- HttpOnly and Secure cookies
- review of all places where stored input is rendered
- safe logging and monitoring
- security testing before release

## What I should never do outside a lab

I should never test stored XSS on public websites, university systems, company systems, real accounts, or third-party applications without permission.

Stored XSS can affect other users and can create real harm.

This lab is only for local, controlled learning.
