# Request Analysis

## GET /

### What request was sent?

The client sent a GET request to the root path:

GET / HTTP/1.1

This requested the main page of the tiny backend app.

### What response came back?

The server returned 200 OK and an HTML page.

The page included a search form and a login form.

This was different from a static file-only setup because the response was generated by backend code.

## GET /search?q=test

### Where is the q parameter located?

The q parameter is located in the URL query string.

Example:

/search?q=test

The parameter name is q and the value is test.

### How did the server use q?

The backend parsed the query string using urlparse and parse_qs.

Then it read the q value and placed it into the HTML response.

Before returning the value, the app used html.escape(q) to encode special HTML characters.

### What changed when q changed?

When the request was:

GET /search?q=test

the response showed:

You searched for: test

When the request was:

GET /search?q=hello

the response showed:

You searched for: hello

This proves that the backend was reading client-controlled input and using it to build the response.

## POST /login

### Where were username and password sent?

The username and password were sent in the POST request body as form data.

Example structure:

username=student&password=REDACTED

This is different from the GET search request because the values were not placed in the URL query string.

### What changed between valid and invalid login?

For the valid login, the backend returned:

200 OK
Login successful

For the invalid login, the backend returned:

401 Unauthorized
Login failed

This shows that the backend applied authentication logic based on the submitted form fields.

### Why credentials should not be sent over plain HTTP

Plain HTTP is not encrypted.

If credentials are sent over HTTP on a real network, someone who can observe the traffic may be able to read them.

This lab used localhost only, but real login forms should use HTTPS.

## Burp observations

### What was easier to see in Burp than in the browser?

Burp made the raw HTTP requests easier to see.

It showed the method, path, headers, query parameters, and POST body in a direct format.

In the browser, these details are hidden behind the normal user interface unless DevTools or a proxy is used.

### What could be modified?

Burp allowed the request path, query string, headers, and form body to be modified before forwarding the request.

For example, I changed:

GET /search?q=test HTTP/1.1

to:

GET /search?q=modified-by-burp HTTP/1.1

The backend then processed the modified q value.

This shows why server-side validation and output encoding are important.
