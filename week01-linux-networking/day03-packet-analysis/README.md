# Day 3 — Packet Analysis Basics

## Goal

The goal of this lab was to understand how network traffic looks at the packet level using tcpdump and tshark.

I captured ICMP, DNS, and HTTPS traffic and analyzed what information is visible from each protocol.

## Tools used

- tcpdump
- tshark
- ping
- nslookup
- curl

## What I captured

I captured:

- ICMP echo request and echo reply packets generated by ping
- DNS queries and responses for example.com
- HTTPS TCP connection metadata to example.com
- Connectivity test output for multiple targets

## What I learned about ICMP

ICMP is used for network control and diagnostic messages.

In this lab, ping generated ICMP echo requests from my WSL system to 8.8.8.8. The remote host replied with ICMP echo replies. This showed that the destination was reachable and that packets could travel both ways.

From a cybersecurity point of view, ICMP can help analysts test connectivity, but it can also be used by attackers during reconnaissance.

## What I learned about DNS

DNS translates domain names into IP addresses.

In this lab, nslookup generated DNS queries for example.com. The capture showed traffic to port 53 and included A records for IPv4 and AAAA records for IPv6.

DNS is important in cybersecurity because many attacks and malware communications begin with DNS lookups.

## What I learned about HTTPS/TLS

HTTPS uses TLS encryption.

In the packet capture, I could see the remote IP, TCP port 443, packet timing, packet sizes, and TCP flags. I could not read the HTTP headers or page content because TLS encrypted the application data.

This matters because analysts can still learn useful metadata from encrypted traffic even when they cannot see the full content.

## What I learned about TCP vs UDP

TCP creates a connection before exchanging data. It uses a three-way handshake and focuses on reliability.

UDP does not create a full connection first. It is faster and has less overhead, but reliability must be handled by the application if needed.

## Cybersecurity relevance

Packet analysis helps security analysts understand what systems are communicating, which protocols are being used, which ports are involved, and whether traffic behavior looks normal or suspicious.

Even simple captures can reveal useful information during troubleshooting, reconnaissance detection, incident response, and network monitoring.

## Mistakes / difficulties

The hardest part was understanding what traffic is visible and what is hidden when HTTPS/TLS is used.

I also had to be careful not to expose private WSL IPs or unnecessary system details in public evidence files.
